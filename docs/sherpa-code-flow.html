代码层级结构

下面这四层直接对应代码职责，而不是概念化的“平台架构图”。

1. 前端控制台

frontend-next/app/page.tsx

页面本身只做装配。它拉起 SystemOverviewCard、 ConfigPanel、 SessionPanel、 TaskProgressPanel、 LogPanel， 再通过 useSystemQuery、useTasksQuery、 useTaskDetailQuery 周期轮询后端。

2. API 编排层

main.py

FastAPI 负责配置读写、任务提交、任务查询、恢复和停止。真正关键的是 _submit_fuzz_job、 _run_fuzz_job、 _execute_k8s_job： 它们把一个 fuzz 请求拆成多个阶段 job，逐段执行并根据返回结果决定下一段。

3. 工作流状态机

workflow_graph.py

使用 LangGraph StateGraph 注册节点和条件跳转。这里不是固定线性流水线， 而是根据 build_error_kind、run_error_kind、 crash 状态和 coverage 结果来路由到 fix_build、 coverage-analysis、re-build、 fix_crash 等分支。

4. 底层执行器

fuzz_unharnessed_repo.py

真正做 clone、脚手架生成、build、run、seed bootstrap、plateau 检测和 crash 产物整理的代码在这里。 它体现的是“如何跑 fuzz”，而 main.py 和 workflow_graph.py 体现的是“何时跑哪个阶段”。

前端如何提交任务

frontend-next/lib/api/client.ts 里的 submitTask() 会调用 POST /api/task， body 中实际发送的是 jobs[]，每个 job 包含 code_url、budget、max tokens、coverage loop 参数和 build retry 参数。

后端如何切阶段

_run_fuzz_job() 在 while 循环里维护 current_stage。每执行完一个 stage， 就从结果中的 workflow_recommended_next 推导下一段， 直到返回 stop 或空值为止。

k8s worker 怎么落地

_execute_k8s_job() 会生成 manifest，调用 kubectl apply -f - 提交 job，等待完成后从 result_path 读取 JSON，失败则读 error/log tail，并更新 job 状态。

状态存在哪

job_store.py 定义了统一接口，当前主用 PostgresJobStore。每个 job 会被序列化到 payload_json，这样服务重启后仍可恢复任务视图。

如果要一句话向别人解释这个项目，可以这样说： Sherpa 通过前端收集 fuzz 任务参数，FastAPI 把请求存成 job，然后逐阶段派发 k8s worker； 每个阶段真正执行的逻辑由工作流状态机和底层执行器共同决定，最后把任务状态、子任务状态和日志再反馈回前端控制台。
